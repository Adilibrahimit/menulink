01 / THE ANSWER

الإجابة المختصرة

نعم، ممكن. وهي ميزة قاتلة في السوق السعودي تحديداً، لأن Foodics يهيمن على ٧٠٪+ من مطاعم المملكة، وعنده API عام مُوثّق بالكامل.

كل ما يصير عند الزبون "إرسال الطلب" في MenuLink، نظامنا يستطيع فوراً إرسال نفس الطلب لـ Foodics (أو Marn أو Loyverse) عبر API، ومن لحظتها كل شيء يحدث تلقائياً:

  • الطلب يظهر في شاشة الكاشير
  • الفاتورة تطبع في طابعة المطبخ (الطباخ يبدأ التحضير)
  • التذكرة تطبع للكاشير
  • المخزون ينقص تلقائياً
  • تقارير المبيعات تتحدّث

كل هذا في أقل من ٢ ثانية بعد ضغط الزبون على "إرسال". وللمطعم: لا داعي لإعادة كتابة الطلب يدوياً، لا أخطاء تحويل، لا تأخير.

04 / ARCHITECTURE

نمط Adapter · واجهة واحدة، عدة POS

المشكلة: كل POS له API مختلف، حقول مختلفة، أسلوب تسجيل دخول مختلف. كيف نخفي هذا التعقيد عن باقي المنصة؟ الإجابة: نمط Adapter.

Fig 02 · Adapter pattern · one contract, many POS
نواة MenuLink لا تعرف ولا تهتم بأي POS · هي تستدعي POSAdapter فقط · كل adapter يترجم للـ API الخاص بنظامه

الواجهة الموحّدة (TypeScript)

// كل adapter يحقّق هذه الواجهة. النواة لا تهتم بالتفاصيل. interface POSAdapter { name: string; connect(credentials: Credentials): Promise<Session>; pushOrder(order: MenuLinkOrder): Promise<POSOrderId>; syncMenu?(restaurantId: string): Promise<Menu>; disconnect(restaurantId: string): Promise<void>; } // الـ Foodics implementation class FoodicsAdapter implements POSAdapter { name = "foodics"; async pushOrder(order: MenuLinkOrder) { const payload = translateToFoodicsFormat(order); const res = await fetch("https://api.foodics.com/v5/orders", { method: "POST", headers: { Authorization: `Bearer ${token}` }, body: JSON.stringify(payload), }); return res.data.id; } } // الـ registry — نختار حسب نوع المطعم const ADAPTERS = { foodics: new FoodicsAdapter(), marn: new MarnAdapter(), loyverse:new LoyverseAdapter(), }; // الاستخدام const adapter = ADAPTERS[restaurant.pos_type]; await adapter.pushOrder(order); // خلاص. هذا كل الكود.

فائدة هذا النمط: لإضافة POS جديد، نكتب ملف واحد فقط (NewPOSAdapter.ts) ونضيفه في الـ registry. صفر تعديلات على باقي الكود. هذا "extensibility" بمعناه الحقيقي.

05 / RESTAURANT UX

تجربة صاحب المطعم · ٣ ضغطات للربط

أنت سألت "بدون تعقيد، بطريقة واضحة وأوتوماتيكية لصاحب المطعم". هذا الجوهر — لو الربط كان معقّداً، الميزة كلها فاشلة. هذه هي التجربة المستهدفة:

١

صاحب المطعم يفتح إعدادات MenuLink

قسم "ربط نظام الكاشير". يرى قائمة بالأنظمة المدعومة (Foodics, Marn, Loyverse). يضغط على نظامه.

⏱️ ٥ ثواني
٢

تسجيل دخول OAuth (مع Foodics)

يضغط "ربط مع Foodics". يفتح له صفحة Foodics. يسجل دخول. يضغط "أوافق على المشاركة". تم. لا API key يكتبه يدوياً، لا تكوين معقّد.

للأنظمة الأخرى (Marn): يلصق API key واحد من إعدادات Marn عنده. خطوة واحدة فقط.

⏱️ ٢٠ ثانية
٣

استيراد المنيو تلقائياً 🎁

هذه هي السحرية: ما نطلب من صاحب المطعم إعادة كتابة المنيو في MenuLink. نقرأ منيوه من Foodics تلقائياً (الأصناف، الأسعار، الصور)، نُدخلها في MenuLink، ونعرضها له للمراجعة. يضغط "اعتمد" → جاهز.

⏱️ ٢ دقيقة (لمراجعة المنيو)
مجموع الوقت لربط مطعم جديد من الصفر: أقل من ٣ دقائق. قارن مع المنافسين اللي يأخذون ساعات أو أيام لإعداد المنيو يدوياً.

اختبار الربط (للتأكد)

بعد الربط، نعرض زر "إرسال طلب تجريبي". إذا ضغطه:

  • نرسل طلباً وهمياً بقيمة ١ ريال إلى POS
  • الطابعة في المطعم تطبع: "طلب تجريبي — MenuLink — تجاهل"
  • صاحب المطعم يلغيه من POS
  • الآن متأكد ١٠٠٪ إن النظام شغّال

الميزة الذكية · مزامنة المنيو في الاتجاهين

هذه فكرة تجعل MenuLink لا يُقاوَم: صاحب المطعم يدير منيو واحد فقط — في Foodics — وكل شيء يتزامن.

Fig 03 · Two-way menu sync
مزامنة في الاتجاهين عبر Foodics webhooks · لا داعي لمحرر منيو منفصل في MenuLink لمن عندهم Foodics

لماذا هذه ميزة بيعية قوية؟

المنصات المنافسة (مثل Chatfood, Foodbit) تطلب من المطعم إعادة كتابة المنيو في منصتها. هذا عمل ساعات. مع MenuLink + Foodics integration: الإعداد كله ٣ دقائق + المنيو يبقى محدّثاً تلقائياً للأبد. هذا فرق هائل في تجربة العميل.

07 / FALLBACK

وش لو المطعم ما عنده POS؟

نسبة كبيرة من المطاعم الصغيرة وعربات الطعام لا تستخدم أي نظام كاشير. ما نتركهم بدون حل. عندنا ٣ طبقات احتياطية:

الطبقة ١ · واتساب (الأساس · مجاني)

كما هو الحال في v6 الآن. الطلب يصل كرسالة واتساب منسّقة على جوال المطعم. الموظف يقرأها ويبدأ التحضير. هذا الحل يعمل لـ ١٠٠٪ من المطاعم بدون أي إعداد.

الطبقة ٢ · لوحة تحكم على iPad / كمبيوتر (مدفوع)

المطعم يفتح لوحة تحكم MenuLink على جهاز في المطبخ. الطلبات تظهر فيها مباشرة (عبر Realtime). صوت تنبيه عند كل طلب جديد. يضغط زر "طُبع" → يحفظ التاريخ. هذا حل وسط بين الواتساب والـ POS الكامل. مفيد للمطاعم اللي تستقبل ١٠+ طلبات يومياً.

الطبقة ٣ · طابعة حرارية مباشرة (للمتقدمين)

المطعم يشتري طابعة حرارية بسيطة (~٢٠٠ ريال) موصولة بالشبكة المحلية. نظامنا يرسل أمر طباعة مباشر عبر بروتوكول ESC/POS (معيار قديم وموثوق منذ ١٩٩٠). الطلبات تطبع تلقائياً بدون POS. ميزة كاملة بنصف التكلفة.

08 / PRICING

أثر هذه الميزة على التسعير

تكامل POS ميزة عالية القيمة. تستحق سعراً أعلى. هذا هيكل تسعير محدّث مع هذه الميزة:

الخطة السعر/شهر المميزات المناسب لـ
Starter 99 ر قائمة PWA + واتساب فقط عربات طعام، مقاهي صغيرة
Pro 249 ر + حسابات الزبائن + إشعارات + لوحة تحكم + طابعة حرارية مطاعم متوسطة
Business ⭐ 499 ر + تكامل Foodics/Marn + مزامنة المنيو + دفع إلكتروني + دعم أولوية سلاسل ومطاعم كبيرة

الحجة البيعية القوية: "اشترك في الخطة Business بـ ٤٩٩ ريال شهرياً، وستوفر ٢-٣ ساعات يومياً من وقت الكاشير اللي كان يدخل طلبات الويب يدوياً." في معظم المطاعم، هذا يعني الميزة تدفع تكلفتها بنفسها خلال أسبوع واحد.

09 / TIMELINE

متى نبني هذه الميزة؟

تكامل POS ليس للمرحلة ١. هذه ميزة المرحلة ٤ أو ٥. الترتيب الصحيح:

  1. المرحلة ١ (الآن): v6 PWA + واتساب. ابدأ ببيعه.
  2. المرحلة ٢: Next.js + Supabase + Auth + Order history.
  3. المرحلة ٣: لوحة تحكم المطعم + Push notifications.
  4. المرحلة ٤ ⭐: تكامل Foodics (٤-٦ أسابيع). هذا يفتح لك سوق المطاعم الكبيرة.
  5. المرحلة ٥: Marn + Loyverse + طابعة حرارية مباشرة.
لا تبني التكامل قبل أن يطلبه ٣ عملاء على الأقل. هندسياً، الإغراء كبير للبدء به مبكراً. لكن تجارياً، تحتاج أن تتحقق من قيمة الميزة من العملاء قبل استثمار ٦ أسابيع تطوير فيها.
