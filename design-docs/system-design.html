01 / PRINCIPLES

قبل الكود · المبادئ الأربعة

كل قرار في هذه المعمارية يخدم أربعة مبادئ. لو حصل تعارض بين فكرة جديدة وأي مبدأ منها، المبدأ يفوز.

"الكود الذي لا يُكتب لا يحتوي على أخطاء." — مبدأ هندسي قديم. كل خدمة نختار عدم بنائها هي خدمة لا يمكن أن تنهار.

المبدأ ١ · لا backend مخصّص

لا نكتب خادم Node.js أو Python أو أي شيء يحتاج صيانة. نستخدم Supabase كـ Backend-as-a-Service. كل ما نحتاجه (قاعدة بيانات، تسجيل دخول، تخزين صور، تحديث لحظي) موجود في خدمة واحدة. السبب: لا يوجد كود خادم = لا توجد أخطاء خادم.

المبدأ ٢ · Stateless Frontend

Next.js على Vercel يخدم HTML/CSS/JS فقط. ما يخزن شيء محلياً. لو فشل خادم Vercel، سيرفر آخر يلتقطه فوراً. Vercel يدير CDN عالمي ويوزّع الحمل على ٢٠+ دولة تلقائياً.

المبدأ ٣ · Data في مكان واحد

كل البيانات (المطاعم، الزبائن، الطلبات، المنيو، الصور) في Supabase. لا نسخ متعددة. لا تزامن. أي تغيير يحصل في مكان واحد ويظهر فوراً لكل من يتصفح. هذا يقضي على ٩٠٪ من بُغ الـ data sync.

المبدأ ٤ · افشل بأمان (Fail-Safe)

لو سقطت الإشعارات، الطلب لا يزال يصل عبر واتساب. لو سقطت الخريطة، العنوان النصي يكفي. لو سقط Supabase، PWA المُثبّت يعرض القائمة من الكاش. كل ميزة تنحط فوق الأساسية، لا تحلّ مكانها.

02 / ARCHITECTURE

المعمارية العامة

أربع طبقات. كل طبقة لها مسؤولية واحدة. كل طبقة قابلة للاستبدال بدون لمس الطبقات الأخرى.

Fig 01 · High-level system architecture
٤ طبقات منفصلة · ٤ خدمات أساسية فقط · لا backend مخصّص · كل خدمة لها بديل لو سقطت

الفكرة المهمة: أنت لا تبني الـ Backend. أنت تبني فقط الـ Frontend (Next.js) وتُهيّئ قاعدة البيانات في Supabase. كل التعقيد (الأمان، التوسع، النسخ الاحتياطية، الـ scaling) Supabase يتولاه. هذا هو السبب الأول لكون الكود "ما يخرب".

المكدّس التقني · ولماذا

كل أداة هنا اختيرت بعد التفكير في البديل. الأخضر = خطة مجانية تكفي للبداية. الأحمر = ندفع فيها لاحقاً عند التوسع.

04 / FLOW · ORDER

تدفق الطلب · من السلة إلى المطعم

هذا أهم تدفق في المنصة كلها. عشر خطوات، كل خطوة في خادم مختلف، لكن من ناحية الزبون: نقرتين فقط.

Fig 02 · Order placement sequence
الزبون يرى نقرتين · النظام يقوم بـ ١١ خطوة في الخلفية · لا شيء منها يحتاج خادم نكتبه نحن

لماذا واتساب + Supabase معاً؟

Supabase يحفظ الطلب في قاعدة البيانات (للتحليلات، التاريخ، إعادة الطلب). واتساب يوصّل التنبيه الفوري للمطعم على جوال الموظف. قناتان مستقلتان: لو سقطت قاعدة البيانات، الواتساب يصل. لو سقط الواتساب، الطلب في قاعدة البيانات يظهر على لوحة المطعم. هذا هو "Fail-safe".

07 / MULTI-TENANT

Multi-Tenant · كود واحد، مطاعم كثيرة

هذا أهم سحب في المنصة. نفس الكود يقدّم تجارب مختلفة كلياً لكل عميل، عبر قراءة config المطعم وتطبيقه على الـ UI.

Fig 05 · Multi-tenant routing & theming
٣ subdomains · نفس الـ Next.js app · ٣ تجارب مختلفة كلياً. لإضافة عميل جديد: صف واحد في جدول restaurants · ٣٠ دقيقة بدل ٣٠ ساعة.

الكود الفعلي (مبسّط)

// app/[tenant]/page.tsx
export async function getServerSideProps({ params }) {
  const tenant = await supabase
    .from('restaurants')
    .select('*, menu_categories(*, menu_items(*))')
    .eq('slug', params.tenant)
    .single();
  return { props: { tenant } };
}

export default function MenuPage({ tenant }) {
  return (
    <ThemeProvider theme={tenant.theme}>
      <Menu data={tenant.menu_categories} />
    </ThemeProvider>
  );
}

١٠ أسطر فقط من الكود تقدّم كل عميل في المنصة. اللوحة الجمالية والمنيو يأتيان من Supabase في كل طلب.

08 / ROBUSTNESS

٧ أسباب تجعل هذا الكود لا يخرب

"ما يخرب" ليست شعار. هي نتيجة قرارات معمارية محددة. هذه ٧ منها:

1

صفر كود خادم نكتبه

كل المنطق في الفرونت أو في Supabase RLS (Row Level Security). لا يوجد server.js يمكن أن يقع. لو حصلت مشكلة في Supabase، فريقهم من ٢٠٠ مهندس يحلّها قبل أن نلاحظ نحن.

2

PostgreSQL هي قاعدة البيانات

PostgreSQL أُطلقت ١٩٩٦. تستخدمها Instagram, Apple, Spotify, Saudi Aramco. مرّت بـ ٣٠ سنة من الاختبار في العالم الحقيقي. أي مشكلة تواجهها، شخص آخر حلّها قبلك.

3

كل ميزة لها بديل احتياطي

الطلب عبر واتساب و قاعدة البيانات. الموقع عبر GPS و العنوان النصي. الإشعارات Push و SMS. لو سقطت أي قناة، الأخرى تكفي.

4

الكاش متعدد المستويات

Vercel CDN يخدم الـ HTML من أقرب مدينة جغرافياً. الـ PWA يحفظ القائمة في جوال الزبون. Supabase يستخدم connection pooling. الفرونت يفتح في ٠.٥ ثانية حتى لو سقط Supabase (سيعرض النسخة المحفوظة).

5

Row Level Security (RLS)

كل عميل (مطعم) يرى فقط بياناته الخاصة. هذه القاعدة مفروضة على مستوى قاعدة البيانات، ليس على مستوى الكود. حتى لو كتبنا استعلام خاطئ، PostgreSQL يرفضه. مستحيل أن يرى مطعم بيانات مطعم آخر.

6

النسخ الاحتياطي التلقائي

Supabase ينسخ قاعدة البيانات تلقائياً كل يوم. الخطة المدفوعة تحتفظ بنسخ لـ ٣٠ يوم. أي خطأ نرتكبه، يمكن استرجاع الحالة قبله بـ نقرة واحدة. لا "Oops حذفنا كل البيانات" في هذه المعمارية.

7

التوسع التلقائي

لو ١٠ مطاعم اشتغلت في نفس اللحظة، Vercel يضيف خوادم تلقائياً. Supabase يقاسم الحمل على instances متعددة. لا نحتاج "ساعة الذروة" للقلق. التطبيق يتعامل مع زيادة الحمل بدون أي تدخل منا.

السبب الحقيقي لكون الكود "ما يخرب": أقل كود ممكن. الكود الذي لا تكتبه لا يحتاج اختباراً، لا تصحيحاً، لا صيانة، لا قلقاً في الليل.
09 / ECONOMICS

التكلفة الحقيقية والتوسع

هذا الجدول يوضّح متى تدفع وكم. الخطوط الخضراء = الخطط المجانية تكفي.

سيناريوهات حقيقية

🌱 السيناريو ١ · ٥ مطاعم أوّليّة

٢٠٠ زبون فعّال شهرياً، ٥٠٠ طلب شهرياً. التكلفة: ٧ ريال شهرياً (نطاق فقط). كل شيء آخر على الخطط المجانية. الإيرادات لو سعرت ١٠٠ ريال لكل مطعم: ٥٠٠ ريال شهرياً. صافي: ٤٩٣ ريال.

🚀 السيناريو ٢ · ٥٠ مطعم نشط

٥٬٠٠٠ زبون، ١٥٬٠٠٠ طلب شهرياً، ٢٬٠٠٠ تسجيل دخول جديد. التكلفة: ~٥٥٠ ريال شهرياً (كل الخطط المدفوعة + SMS). الإيرادات: ٢٠٠ ريال × ٥٠ = ١٠٬٠٠٠ ريال. صافي: ٩٬٤٥٠ ريال شهرياً.

📈 السيناريو ٣ · ٢٠٠ مطعم

هذا حد لا يحتاجك صاحب شركة وحدك. عند هذا الحجم، تستأجر مطورين + خدمة عملاء + مدير علاقات. التكلفة الفنية ~٢٠٠٠ ريال شهرياً. الإيرادات: ٤٠٬٠٠٠+ ريال. الفرق يذهب للموظفين والربح.

الخطوة التالية

هذا التصميم جاهز للتنفيذ. الفرق بين الفهم و البناء هو القرار:
تبني بنفسك (٣-٦ أشهر تعلّم) أم توظّف مطوّر (٢-٣ أشهر تنفيذ، ٢٬٠٠٠-٥٬٠٠٠ ريال شهرياً)؟

ابدأ بـ ٥ مطاعم على PWA الحالي. اختبر السوق. اجمع الـ feedback. ثم ابني.
