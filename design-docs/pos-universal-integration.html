السؤال الكبير: هل ممكن template واحد يخدم كل أنظمة POS؟ الإجابة الصادقة: لا، لكن معمارية متدرّجة ٥ مستويات تغطي ٩٥٪ من الحالات. والـ POS اللي تبنيه بنفسك يدخل في أسهل مستوى.

01 / REALITY CHECK

الحقيقة الكاملة قبل الكود

لا يوجد "template سحري" يدمج مع كل POS في العالم لأن:

  • كل POS له API مختلف (لو عنده API أصلاً)
  • أساليب المصادقة تتفاوت (API key، OAuth، JWT، توقيع رقمي)
  • أشكال البيانات مختلفة (JSON، XML، حتى ملفات نصية)
  • منطق الأعمال يختلف (بعضها يطلب SKU، بعضها يطلب item name، بعضها لا يقبل تعديل)
  • نسبة كبيرة من أنظمة POS القديمة في السعودية ما عندها API أصلاً
الحل ليس "template واحد" — الحل "سلّم متدرّج" من ٥ مستويات. كل مستوى يغطي فئة من أنظمة POS. مجتمعةً، تغطي كل ما تحتاج.
02 / THE 5-TIER LADDER

السلّم: ٥ مستويات لتغطية أي POS

كل مستوى أصعب قليلاً من السابق، لكن يحلّ مشاكل أكثر. ابدأ من الأعلى، انزل حسب الحاجة.

1
EASIEST · OUR CODE

POS تبنيه بنفسك (مثل .NET POS تبعك)

أنت تتحكم بالكود ١٠٠٪. تضيف REST API endpoint في الـ POS، وMenuLink يرسل الطلبات لها. هذا الأسهل والأقوى — لا قيود من شركة خارجية، لا حاجة لإذن، لا توثيق محدود.

أمثلة: .NET POS تبنيه · أي POS مفتوح المصدر · أي شيء تطوّره أنت

2
EASY · NATIVE API

POS تجاري له REST API عام

نكتب adapter يتحدث مع API الخاص بهم. أحياناً OAuth، أحياناً API key. توثيقهم يخبرنا بالحقول والـ endpoints. عمل ١-٢ أسبوع لكل POS جديد.

أمثلة: Foodics ⭐ · Marn · Loyverse · Square · Toast · Lightspeed

3
MODERATE · WEBHOOK

POS يقبل webhooks فقط

POS لا يعطي API كامل، لكن يقدر يستقبل HTTP POST. نرسل الطلب لـ URL يحدده المطعم، الـ POS يستقبله ويعالجه. أبسط لكن أقل تحكم.

أمثلة: بعض أنظمة POS الإقليمية الناشئة · حلول مخصّصة

4
FALLBACK · DIRECT PRINT

POS ما عنده API — نطبع مباشرة على طابعة المطبخ

نتجاوز POS كلياً. نتصل بطابعة حرارية في المطبخ عبر بروتوكول ESC/POS (معيار منذ ١٩٩٠ في كل طابعة حرارية). الطابعة تطبع الطلب → الطباخ يبدأ التحضير → الكاشير يدخل الطلب يدوياً في POS لاحقاً.

أمثلة: أنظمة POS قديمة بدون API · مطاعم بـ Excel · ٥٠٪+ من المطاعم الصغيرة

5
LAST RESORT · BRIDGE APP

برنامج جسر (Bridge) يُثبَّت على كمبيوتر المطعم

آخر حل: نطوّر تطبيق Windows صغير يُثبَّت في المطعم. يسحب الطلبات من MenuLink API كل ٣٠ ثانية، ثم يدخلها في POS عبر:

  • إدخال مباشر لقاعدة بيانات POS (لو نعرف schema)
  • محاكاة لوحة المفاتيح (يكتب الطلب في POS تلقائياً)
  • استدعاء COM أو DLL خاص بـ POS

هذا حل تقني صعب — نستخدمه فقط لعميل كبير ضروري ولا يمكن إقناعه بترقية POS.

القاعدة الذهبية: ابدأ بـ Tier 1 (POS الخاص بك .NET) أو Tier 2 (Foodics). غطّ ٨٠٪ من السوق. لا تبني Tier 4 أو 5 إلا لمّا يطلبها عميل ضروري وقيمته تستحق التطوير الإضافي.

03 / DECISION TREE

أي مستوى لأي POS؟

ضع POS أي مطعم في هذه الشجرة، تعرف فوراً وش الحل المناسب:

❓ هل أنت طوّرت الـ POS بنفسك؟
✓ نعم → استخدم Tier 1 · أضف REST API endpoint في .NET POS
✗ لا → السؤال التالي
❓ هل POS التجاري عنده REST API عام موثّق؟
✓ نعم → Tier 2 · اكتب adapter (Foodics, Marn, إلخ)
✗ لا → السؤال التالي
❓ هل POS يقبل HTTP webhooks خارجية؟
✓ نعم → Tier 3 · أرسل الطلبات للـ URL
✗ لا → السؤال التالي
❓ هل المطعم عنده طابعة حرارية في المطبخ؟
✓ نعم → Tier 4 · ارسل ESC/POS مباشرة للطابعة
✗ لا → السؤال التالي
❓ هل قيمة العميل تستحق ٤ أسابيع تطوير؟
✓ نعم → Tier 5 · بناء Bridge App
✗ لا → استخدم واتساب فقط · يكفي
04 / .NET POS

تكامل MenuLink مع POS تبنيه بـ .NET

الخبر الجيد: لأنك تطوّر POS بنفسك بـ .NET، التكامل أسهل بكثير من Foodics. لا تنتظر إذن من شركة، لا قيود توثيق، لا rate limits. تتحكم في كل شيء. .NET يحتوي أصلاً على كل شيء تحتاجه لبناء API احترافي.

المعمارية: كيف يتحدث MenuLink مع POS تبعك

Fig 01 · MenuLink → Your custom .NET POS
MenuLink لا يهتم إن POS كان Foodics أو .NET تبعك أو أي شيء آخر. الواجهة POSAdapter هي اللي تترجم لكل نوع.

OrdersController.cs
// ASP.NET Core 8 · أضف هذا في مشروع POS الموجود [ApiController] [Route("api/orders")] [Authorize(AuthenticationSchemes = "ApiKey")] public class OrdersController : ControllerBase { private readonly IOrderService _orderService; private readonly IPrinterService _printer; public OrdersController(IOrderService orderService, IPrinterService printer) { _orderService = orderService; _printer = printer; } // MenuLink يستدعي هذا الـ endpoint لإرسال طلب جديد [HttpPost] public async Task<IActionResult> CreateOrder([FromBody] MenuLinkOrderDto dto) { // 1. تحقّق من البيانات if (!ModelState.IsValid) return BadRequest(ModelState); // 2. حوّل من شكل MenuLink لشكل POS الداخلي var order = new Order { CustomerName = dto.CustomerName, CustomerPhone = dto.CustomerPhone, OrderType = dto.OrderType, // delivery/pickup/dinein Source = "MenuLink", ExternalId = dto.OrderId, // reference for tracking Items = dto.Items.Select(i => new OrderItem { ProductCode = i.ProductCode, Quantity = i.Quantity, UnitPrice = i.Price, Notes = i.Notes }).ToList(), DeliveryAddress = dto.Address, Latitude = dto.Latitude, Longitude = dto.Longitude, Status = OrderStatus.New }; // 3. احفظ في قاعدة بيانات POS await _orderService.CreateAsync(order); // 4. اطبع التذكرة في المطبخ تلقائياً await _printer.PrintKitchenTicketAsync(order); // 5. ارجع التأكيد لـ MenuLink return Ok(new { success = true, posOrderId = order.Id, estimatedReadyAt = DateTime.Now.AddMinutes(20) }); } }

الميزة الكبيرة لـ SignalR: ما تحتاج كمبيوتر POS يكون عنده IP عام (public IP). يفتح اتصال خارج من جوّا، يبقى مفتوح. يعمل من أي مكان حتى وراء راوتر منزلي. هذا أمان وراحة في آن واحد.

05 / BEST PRACTICES

أفضل الممارسات لتكامل آمن وموثوق

١. Idempotency — منع التكرار

لو MenuLink أرسل طلباً وسقط الاتصال قبل وصول الرد، MenuLink يعيد الإرسال. لازم POS يكون ذكي ولا يحفظ الطلب مرتين. الحل:

منطق Idempotency
// قبل حفظ الطلب، تحقّق إذا كان موجوداً var existing = await _orderService .GetByExternalIdAsync(dto.OrderId); if (existing != null) { // طلب مكرر — ارجع نفس الرد، لا تعيد الحفظ ولا الطباعة return Ok(new { success = true, posOrderId = existing.Id }); } // لو ما كان موجوداً، احفظ واطبع كالعادة

٢. Retry مع Exponential Backoff

في جانب MenuLink، لو فشل الإرسال، نعيد المحاولة بفترات متباعدة: ١ ثانية، ٢ ثانية، ٤ ثوان، ٨ ثوان. لو فشل بعد ٤ محاولات، نرسل تنبيه لصاحب المطعم عبر واتساب: "طلب لم يصل لـ POS — تحقّق يدوياً". صفر طلبات ضائعة.

٣. الـ Order Status Loop

MenuLink ↔ POS اتصال في الاتجاهين. لما يحدّث الكاشير حالة الطلب (تم القبول → قيد التحضير → جاهز → خرج للتوصيل)، الـ POS يخبر MenuLink. الزبون يشوف التحديثات على الـ PWA لحظياً.

٤. Menu Mapping

الصنف "بروستد عادي" في MenuLink قد يكون "PROD-001" في POS تبعك. عند الإعداد، تحدد المطابقة مرة وحدة. بعدها MenuLink يرسل المعرفات الداخلية بدل الأسماء، وPOS يطبع المسمّى الذي يفهمه الكاشير.

كل هذه الممارسات سهلة في .NET لأنك تتحكم بالكود. أصعب جزء في تكامل Foodics هو القيود على التوثيق وعدم القدرة على تعديل سلوكه. مع POS تبنيه بنفسك، أي مشكلة تظهر تحلّها فوراً في الكود.

06 / STRATEGY

الاستراتيجية: تجمع بين المنصة والـ POS

🎯 فرصة كبيرة لك تحديداً

لأنك تطوّر POS بـ .NET وتطوّر MenuLink في نفس الوقت، عندك ميزة فريدة: تقدر تبيع الباقة الكاملة للمطاعم — POS + Menu أونلاين + توصيل + تكامل تام.

المنافسون (Foodics + أي menu link خارجي) يكلّفون المطعم ٢٬٠٠٠+ ريال شهرياً. أنت تقدر تعرض الباقة بـ ٧٠٠-١٠٠٠ ريال وتربح أكثر منهم لأن:

  • لا تدفع نسبة لشركة POS خارجية
  • تتحكم بكل تجربة المطعم — تطوير أسرع، صيانة أبسط
  • كل ميزة جديدة تبنيها مرة في الـ POS تنعكس فوراً على MenuLink

التسعير المقترح للباقة المدمجة

الباقة السعر/شهر المكوّنات
MenuLink فقط 249 ر PWA + واتساب + لوحة تحكم
POS .NET فقط 399 ر كاشير + طابعات + تقارير + مخزون
⭐ الباقة الكاملة 599 ر POS + MenuLink + تكامل تلقائي + دعم أولوية
الفائدة الحقيقية: عميل واحد، سعر واحد، فاتورة واحدة. المطعم ما يحتاج يتعامل مع شركتين، يدفع لاثنين، يتصل دعمين فني. أنت المرجع الوحيد لكل احتياجات المطعم التقنية. هذا تماسك يصعب على المنافسين تقليده.
