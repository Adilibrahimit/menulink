Must-fix before implementation

  1. Write the CHECK-widen DDL. DROP CONSTRAINT IF EXISTS qr_scan_events_source_type_check … ADD … CHECK (source_type IN (…, 'menu')). Verify name with \d first. Test on a branch.
  2. Add export const dynamic = "force-dynamic" to /m/[slug]/page.tsx — kills the silent-zero failure.
  3. Rename the metric unique_visitors → daily_devices (or footnote). Never market as analytics-grade.
  4. Make the view security_invoker = true, don't grant to anon, comment the requireOps() gating.
  5. Add composite index (restaurant_id, scanned_at, source_type) on qr_scan_events.
  6. Daily boundary → Asia/Riyadh in the salt and Ops bucketing (UTC splits the dinner rush across 2 days).
  7. Resolve the double-write — page.tsx logs 'menu' for direct/bookmarked hits only; exclude resolved-QR (/q/[code]) traffic so the two paths never double-count.
  8. Filter bot/preview UAs (Googlebot, WhatsApp/Telegram/Slack link-preview crawlers) before insert — WhatsApp is the main share channel.
  9. PDPL retention policy — e.g. a 90-day deletion job for ip_hash/UA rows.
